# TDD コマンド

Red → Green → Refactor の対話的 TDD サイクルを実行します。

## 使い方

```
/tdd <実装したい機能の説明>
```

## プロンプト

以下の TDD サイクルで `$1` を実装してください：

### Phase 1: Red（失敗するテストを書く）

1. 要件を分析し、テストケースを設計します：
   - 正常系（ハッピーパス）
   - 異常系（エラーケース）
   - 境界値

2. プロジェクトのテストフレームワークに合わせてテストファイルを作成します
   - AAA パターン（Arrange / Act / Assert）で記述
   - テスト名は「何をしたら何が起きるか」を明確に

3. テストを実行して **失敗** を確認します

**ここでユーザーに確認**: テストケースの内容が適切か確認してもらいます。

### Phase 2: Green（テストを通す最小限の実装）

1. テストを通すための **最小限の実装** を行います
   - 過度な設計は避ける
   - まずは動くコードを書く

2. テストを実行して **成功** を確認します

3. すべてのテストが通ることを確認します

### Phase 3: Refactor（リファクタリング）

1. コードの品質を改善します：
   - 重複の除去
   - 命名の改善
   - 構造の整理

2. リファクタリング後もテストが通ることを確認します

### サイクルの繰り返し

1つのテストケースが完了したら、次のテストケースに進みます。
各サイクルの完了時にユーザーに進捗を報告し、次に進むか確認します。

### 完了レポート

```markdown
## TDD 完了レポート

### 実装した機能
[機能の概要]

### テストケース一覧
1. ✅ [テスト名] — [説明]
2. ✅ [テスト名] — [説明]

### ファイル変更
- `src/path/to/impl` — 実装ファイル
- `src/path/to/impl.test` — テストファイル

### カバレッジ
[カバレッジ情報（取得可能な場合）]
```
