# Git ワークフロー規約

## コミットメッセージ

Conventional Commits に準拠:

```
<type>: <日本語での簡潔な説明>

- 詳細1
- 詳細2
```

### Type 一覧

| Type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 機能変更を伴わないコード改善 |
| `test` | テストの追加・修正 |
| `style` | フォーマット変更（機能影響なし） |
| `chore` | ビルド・設定・依存関係の変更 |
| `perf` | パフォーマンス改善 |

### ルール
- 1コミット = 1つの論理的変更
- コミットメッセージは現在形で記述
- 本文は「何を」「なぜ」を説明

## ブランチ命名規則

```
<type>/<簡潔な説明>
```

| プレフィックス | 用途 |
|--------------|------|
| `feature/` | 新機能 |
| `fix/` | バグ修正 |
| `refactor/` | リファクタリング |
| `docs/` | ドキュメント |
| `chore/` | 設定・保守作業 |

## Pull Request 規約

### タイトル
- Conventional Commits の type を接頭辞にする
- 70文字以内で簡潔に

### 本文の構成
```markdown
## 概要
[変更の目的と背景を1-3文で]

## 変更内容
- [変更点1]
- [変更点2]

## テスト方法
- [ ] テスト1
- [ ] テスト2
```

## 禁止事項

- `main` / `master` への直接コミット禁止
- `--force` プッシュは共有ブランチでは禁止
- コミットメッセージに機密情報を含めない
