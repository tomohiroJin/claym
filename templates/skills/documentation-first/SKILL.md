# ドキュメント駆動開発

実装前にドキュメントを作成し、仕様を明確にしてから開発を進めます。

## 手順

### 1. 要件の整理

- 機能の目的と背景を明文化する
- ユースケースを列挙する
- 非機能要件（パフォーマンス、セキュリティ等）を整理する
- 制約条件を明確にする

### 2. API 仕様の作成（API の場合）

- エンドポイント一覧を定義する
- リクエスト・レスポンスの形式を記述する
- エラーケースとレスポンスを定義する
- 認証・認可の要件を記述する

### 3. README / 利用ガイドの作成

- 機能の概要を記述する
- セットアップ手順を記述する
- 基本的な使い方を記述する
- 設定項目を一覧化する

### 4. 設計ドキュメントの作成（必要に応じて）

- アーキテクチャ概要図を作成する
- データモデルを定義する
- コンポーネント間の依存関係を整理する
- 重要な設計判断とその理由を記録する

### 5. レビューとフィードバック

- ドキュメントをチームでレビューする
- フィードバックを反映する
- 仕様が合意されたら実装を開始する

### 6. ドキュメントと実装の同期

- 実装中に仕様変更があればドキュメントを更新する
- 実装完了後にドキュメントと実装の整合性を確認する
- コード内のコメントはドキュメントと一貫性を保つ

## ドキュメント種別

| 種別 | タイミング | 目的 |
|------|-----------|------|
| API 仕様 | 実装前 | インターフェースの合意 |
| README | 実装前/中 | 利用方法の案内 |
| 設計ドキュメント | 実装前 | アーキテクチャの合意 |
| CHANGELOG | 実装後 | 変更履歴の記録 |

## 注意事項

- ドキュメントは簡潔に保つ（詳しすぎるドキュメントは更新が困難）
- ドキュメントと実装の乖離を防ぐ仕組みを作る
- 自動生成できるドキュメントは手書きしない
- ドキュメントもコードと同様にバージョン管理する
