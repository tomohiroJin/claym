# API 設計

REST API の設計ベストプラクティスに基づいて、一貫性のある使いやすい API を設計します。

## 設計手順

### 1. リソースの特定

- 操作対象のリソース（名詞）を特定する
- リソース間の関係（1:N、N:M）を整理する
- リソースの階層構造を決定する

### 2. エンドポイント設計

- リソース名は複数形の名詞を使用する（`/users`、`/orders`）
- ネストは2階層までに制限する（`/users/{id}/orders`）
- 動詞はHTTPメソッドで表現する（エンドポイントに動詞を含めない）

| 操作 | メソッド | パス例 |
|------|---------|--------|
| 一覧取得 | GET | `/users` |
| 単体取得 | GET | `/users/{id}` |
| 作成 | POST | `/users` |
| 全体更新 | PUT | `/users/{id}` |
| 部分更新 | PATCH | `/users/{id}` |
| 削除 | DELETE | `/users/{id}` |

### 3. リクエスト・レスポンス設計

- JSON 形式を標準とする
- フィールド名は camelCase を使用する
- 日時は ISO 8601 形式（`2025-01-01T00:00:00Z`）
- ページネーションはクエリパラメータで制御する

### 4. ステータスコードの使い分け

| コード | 用途 |
|--------|------|
| 200 | 成功（GET、PUT、PATCH、DELETE） |
| 201 | 作成成功（POST） |
| 204 | 成功（レスポンスボディなし） |
| 400 | リクエスト不正 |
| 401 | 認証エラー |
| 403 | 認可エラー |
| 404 | リソースが見つからない |
| 409 | 競合（重複作成等） |
| 422 | バリデーションエラー |
| 500 | サーバー内部エラー |

### 5. エラーレスポンス設計

統一されたエラーレスポンス形式を定義する:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力値が不正です",
    "details": [
      {
        "field": "email",
        "message": "有効なメールアドレスを入力してください"
      }
    ]
  }
}
```

### 6. バージョニング

- URL パスにバージョンを含める（`/v1/users`）
- 破壊的変更時にバージョンを上げる
- 旧バージョンの廃止は十分な移行期間を設ける

## 設計チェックリスト

- [ ] リソースが適切に名詞で命名されている
- [ ] HTTP メソッドが正しく使い分けられている
- [ ] ステータスコードが適切に設定されている
- [ ] エラーレスポンスが統一されている
- [ ] ページネーションが実装されている（一覧系）
- [ ] 認証・認可が考慮されている
