# 安全なリファクタリング

テストによる保護の下で、段階的にコード構造を改善します。

## 手順

### 1. 準備: テストで現状を保護する

- 対象コードの既存テストを確認する
- テストが不足している場合、**リファクタリング前に**テストを追加する
- 全テストが通ることを確認する
- テストカバレッジを記録する

### 2. 計画: リファクタリングの方針を決める

- 改善目標を明確にする（可読性、保守性、パフォーマンス等）
- 変更のスコープを限定する（一度に全てを変えない）
- 影響範囲を特定する
- 小さなステップに分割する

### 3. 実行: 小さなステップで進める

各ステップで以下を繰り返す:

1. **1つの変更**を行う（リネーム、関数抽出、移動等）
2. テストを実行して**全テストが通ることを確認**する
3. 問題があれば即座に元に戻す
4. 成功したらコミットする

### 4. 検証: 品質を確認する

- 全テストがパスすることを確認する
- テストカバレッジが下がっていないことを確認する
- リンター・型チェックがパスすることを確認する
- 外部からの振る舞いが変わっていないことを確認する

## よく使うリファクタリングパターン

| パターン | 目的 |
|---------|------|
| 関数の抽出 | 長い関数を小さな関数に分割 |
| 変数のリネーム | 意図が伝わる名前に変更 |
| 条件式の簡略化 | 早期リターンでネストを削減 |
| 重複の排除 | DRY 原則の適用 |
| パラメータオブジェクトの導入 | 引数が多い関数の整理 |
| ガード節の導入 | 事前条件を明示 |

## 注意事項

- リファクタリングと機能変更を同時に行わない
- テストなしでリファクタリングしない
- 大きな変更は小さなステップに分割する
- 各ステップの後にテストを実行する
- 問題が起きたら即座にリバートする
