# 体系的デバッグ

仮説駆動のアプローチで、効率的に問題の原因を特定し解決します。

## デバッグ手順

### 1. 問題の明確化

- 期待される動作と実際の動作を明確にする
- エラーメッセージ・スタックトレースを確認する
- 問題が発生する条件を特定する（入力値、環境、タイミング等）
- 問題がいつから発生しているか確認する

### 2. 問題の再現

- 問題を確実に再現できる手順を確立する
- 最小再現ケースを作成する
- 再現テストを書く（修正後の回帰テストにもなる）

### 3. 仮説の立案

- 問題の原因として考えられる仮説を列挙する
- 各仮説の可能性を評価する
- 最も可能性の高い仮説から検証する

### 4. 仮説の検証

- ログ出力やデバッガで実行状態を確認する
- 二分探索で問題箇所を絞り込む
- 入力値を変えて振る舞いの変化を観察する
- 関連するコードの変更履歴を確認する

### 5. 修正と検証

- 原因が特定できたら、最小限の修正を行う
- 再現テストが通ることを確認する
- 既存テストが全てパスすることを確認する
- 類似の問題がないか確認する

### 6. 再発防止

- 回帰テストを追加する
- 根本原因に対する対策を検討する
- 必要に応じてドキュメントを更新する

## デバッグ技法

| 技法 | 用途 |
|------|------|
| 二分探索 | 問題箇所を半分ずつ絞り込む |
| ログ出力 | 実行時の状態を確認する |
| 差分分析 | 動作する状態との差分を確認する |
| ラバーダック | 問題を他者に説明して気づきを得る |
| 最小再現 | 問題を最小限のコードで再現する |

## 注意事項

- 推測で修正しない（仮説を立て、検証してから修正する）
- 一度に複数の変更をしない（原因の特定が困難になる）
- 修正は最小限にする（関連コードの改善は別タスクで行う）
- デバッグ用のログやコードは最終的に削除する
