# 検証ループ

コード変更後に体系的な検証を行い、品質を保証します。

## 検証手順

### 1. 静的解析

- リンターを実行してコードスタイルの違反がないか確認する
- 型チェックを実行する（TypeScript: `tsc --noEmit`、Python: `mypy` 等）
- フォーマッターを実行する

### 2. ユニットテスト

- 変更に関連するテストを実行する
- 全テストスイートを実行する
- テストカバレッジを確認する（新規コードが 80% 以上カバーされているか）

### 3. ビルド確認

- ビルドが成功するか確認する
- ビルド時の警告がないか確認する
- ビルド成果物が正しく生成されるか確認する

### 4. 統合テスト（該当する場合）

- API テストを実行する
- E2E テストを実行する
- 依存サービスとの結合を確認する

### 5. 結果の判定

- 全チェックが通過 → 変更を確定する
- 失敗がある → 原因を特定して修正し、再度検証する
- 新たな問題が発見 → 問題を記録し、対応方針を決める

## 検証チェックリスト

- [ ] リンター: エラーなし
- [ ] 型チェック: エラーなし
- [ ] ユニットテスト: 全件パス
- [ ] ビルド: 成功
- [ ] カバレッジ: 基準値以上

## 注意事項

- 検証はコミット前に必ず実行する
- CI/CD で実行されるチェックをローカルで先に確認する
- 失敗時は原因を特定してから修正する（闇雲に変更しない）
- 警告も無視せず、可能な限り解消する
