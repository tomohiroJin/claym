# コードレビュー

構造化されたチェックリストに従って、コードの品質を多角的に評価します。

## レビュー手順

### 1. 変更の全体像を把握する

- 変更の目的と背景を理解する
- 影響範囲を確認する（変更ファイル数、依存関係）
- コミット履歴から変更の意図を読み取る

### 2. 正確性の確認

- ビジネスロジックが要件を満たしているか
- エッジケースが適切に処理されているか
- エラーハンドリングが適切か
- 戻り値・引数の型が正しいか

### 3. コード品質の評価

#### 可読性
- 変数名・関数名が意図を表現しているか
- 関数が単一責任の原則に従っているか
- ネストが深すぎないか（早期リターンの活用）
- マジックナンバーが定数化されているか

#### 保守性
- DRY 原則（重複コードがないか）
- 適切な抽象化レベルか（過度な抽象化も問題）
- テストが追加・更新されているか
- ドキュメントが必要に応じて更新されているか

#### パフォーマンス
- 不要なループ・再計算がないか
- 適切なデータ構造が使われているか
- N+1 クエリなどのパフォーマンス問題がないか

### 4. セキュリティの確認

- ユーザー入力のバリデーション
- SQL/NoSQL インジェクション対策
- XSS 対策
- 機密情報のハードコーディングがないか

### 5. フィードバックの記述

- 問題の重要度を明示する（必須修正 / 推奨 / 提案）
- 理由を具体的に説明する
- 可能であれば改善案のコード例を示す
- 良い点も積極的にコメントする

## フィードバック分類

| 分類 | 意味 |
|------|------|
| **必須** | マージ前に修正が必要 |
| **推奨** | 修正が望ましいが必須ではない |
| **提案** | 改善のアイデア（今回の対応は任意） |
| **質問** | 意図や理由の確認 |
| **称賛** | 良い実装・工夫への評価 |
