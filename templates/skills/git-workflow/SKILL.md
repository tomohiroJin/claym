# Git ワークフロー

Conventional Commits とブランチ戦略に基づいた、一貫性のある Git 運用を実践します。

## コミットの作法

### Conventional Commits 形式

```
<type>: <簡潔な説明>

<本文（任意）>
```

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントの変更 |
| `refactor` | 機能変更を伴わないコード改善 |
| `test` | テストの追加・修正 |
| `style` | フォーマット変更（機能影響なし） |
| `chore` | ビルド・設定・依存関係の変更 |
| `perf` | パフォーマンス改善 |

### コミットの原則

- 1コミット = 1つの論理的変更
- コミットメッセージは「何を」「なぜ」を説明する
- 動作する状態でコミットする（壊れた状態をコミットしない）
- 大きな変更は小さなコミットに分割する

## ブランチ戦略

### ブランチ命名規則

```
<type>/<簡潔な説明>
```

| プレフィックス | 用途 |
|--------------|------|
| `feature/` | 新機能 |
| `fix/` | バグ修正 |
| `refactor/` | リファクタリング |
| `docs/` | ドキュメント |
| `chore/` | 設定・保守作業 |

### ブランチの運用

- ブランチは短命に保つ（長くても1-2週間）
- メインブランチから最新を取り込んでから作業する
- 1ブランチ = 1つの目的

## プルリクエスト

### PR の構成

```markdown
## 概要
[変更の目的と背景を1-3文で]

## 変更内容
- [変更点1]
- [変更点2]

## テスト方法
- [ ] テスト1
- [ ] テスト2
```

### PR の原則

- タイトルは70文字以内で簡潔に
- 変更は小さく保つ（レビューしやすい単位）
- 自己レビューしてから提出する
- CI が通ってからレビューを依頼する

## 注意事項

- `main` / `master` への直接コミット禁止
- 共有ブランチでの `--force` プッシュ禁止
- コミットメッセージに機密情報を含めない
- マージ前にコンフリクトを解消する
