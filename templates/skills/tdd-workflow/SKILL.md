# TDD ワークフロー

テスト駆動開発（TDD）の Red-Green-Refactor サイクルに従って、品質の高いコードを段階的に実装します。

## 手順

### 1. Red: 失敗するテストを書く

- 実装したい振る舞いを1つ選ぶ
- その振る舞いを検証するテストを書く
- テストを実行し、**失敗することを確認**する
- テストは具体的で小さくする（1テスト1アサーション）

### 2. Green: テストを通す最小限のコードを書く

- テストを通すために必要な**最小限の実装**を行う
- 過度な設計や将来の要件を先取りしない
- テストを実行し、**全テストが成功することを確認**する

### 3. Refactor: コードを改善する

- テストが通っている状態を維持しながらリファクタリング
- 重複を排除する
- 命名を改善する
- 関数・メソッドの責務を明確にする
- リファクタリング後、テストが全て通ることを確認する

### 4. サイクルを繰り返す

- 次の振る舞いに対して 1-3 を繰り返す
- 各サイクルは小さく保つ（5-10分以内が目安）
- コミットは各 Green/Refactor フェーズで行う

## テスト記述の原則

- **AAA パターン**: Arrange（準備）→ Act（実行）→ Assert（検証）
- **振る舞いベース**: 内部実装ではなく外部から見た振る舞いをテスト
- **独立性**: 各テストは他のテストに依存しない
- **可読性**: テスト名で「何をしたら何が起きるか」を表現

## 注意事項

- テストを書く前にプロダクションコードを書かない
- 一度に複数のテストを追加しない（1つずつ進める）
- テストコードもプロダクションコードと同じ品質基準を適用する
- 外部依存はモック/スタブで分離する
